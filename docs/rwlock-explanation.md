# rwlock.asm 讲解：从互斥到读写锁

## 这个程序演示了什么

两个进程共享**一个全局信号量（初始值=1）**，竞争进入临界区。
信号量确保同一时刻**只有一个进程**在临界区内——这就是互斥（Mutual Exclusion）。

**读写锁的本质就是互斥的扩展**：读者共享、写者互斥。这个 demo 演示了读写锁底层依赖的信号量机制。

## 逐行讲解

```
MOV R1, #0      ; R1 = 0，全局信号量 ID
                ; 系统启动时预创建，所有进程共享

; ========== 每个进程执行 10 轮 ==========

MOV R0, #201    ; 系统调用：sem_wait(R1)
INT 0x80        ; ──→ 如果信号量 > 0：减 1，继续执行
                ; ──→ 如果信号量 = 0：进程阻塞，CPU 让出

MOV R0, #1      ; 系统调用：print
MOV R1, #0x5B   ; 打印 '[' = 进入临界区
INT 0x80

MOV R1, #0      ; 恢复信号量 ID
MOV R0, #202    ; 系统调用：sem_signal(R1)
INT 0x80        ; ──→ 信号量 +1，唤醒一个等待进程

MOV R0, #1      ; 打印 ']' = 退出临界区
MOV R1, #0x5D
INT 0x80
```

## 为什么看到的输出是 [ ] [ ] 交替

两个进程（PID 2 和 PID 3）在 **2 个 vCPU** 上同时运行：

```
时间 →
CPU0: sem_wait(0) → count: 1→0 → 进入 → 打印 [ → sem_signal → count: 0→1 → 打印 ]
CPU1: sem_wait(0) → count: 0   → 阻塞等待 ────────────────────→ count: 1→0 → 进入...
```

**关键观察**：输出中不会出现 `[ [`（两个进程同时进入）。信号量保证了互斥。

## 从信号量到读写锁（向老师讲解的逻辑链）

```
信号量 (semaphore)
  │
  ├── 值=1 → 互斥锁 (mutex) ← 这就是 rwlock.asm 演示的
  │
  └── 值=N → 计数信号量
       │
       └── 读写锁 (rwlock) 的实现依赖:
            ├── 1 个互斥锁 (保护 readcount)
            ├── 1 个信号量 (写者互斥)
            └── 1 个计数器 (readcount = 活跃读者数)

读写锁的伪代码 (C):
  reader() {
    mutex.lock();
    if (++readcount == 1) rw_sem.wait();  // 第一个读者锁住写者
    mutex.unlock();
    // ... 读取 ...
    mutex.lock();
    if (--readcount == 0) rw_sem.signal(); // 最后一个读者释放写者
    mutex.unlock();
  }

  writer() {
    rw_sem.wait();   // 等所有读者退出
    // ... 写入 ...
    rw_sem.signal();
  }
```

**我们的 ISA 没有条件跳转**，所以无法实现 `if (++readcount == 1)` 这样的判断。
但这个 demo 证明了：信号量机制在 genshin-os 中正确工作——
它是构建更复杂同步原语的基础。
